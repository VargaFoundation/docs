Wire format​

The API emits JSON in snake_case by default, matching the reference MLflow REST API and the official Python / Java / R clients. Override with kirka.api.naming=camel_case only for short-term backwards compatibility with legacy consumers.

All date fields are epoch-millis integers. Errors follow the MLflow error body:

{
  "error_code": "RESOURCE_DOES_NOT_EXIST",
  "message": "Experiment 'exp-42' not found",
  "request_id": "8a3f-…"
}

The request_id is the value of the X-Request-Id header (supplied by the caller or generated by Kirka's MDC filter) and is echoed back in the response header, the log lines and every audit row for that request.

Supported MLflow endpoints​

Experiments (/api/2.0/mlflow/experiments)​

create, get, get-by-name, list, search, update, delete, restore, set-experiment-tag (alias set-tag).

list and search accept max_results and page_token; search also accepts a full MLflow filter expression — see Examples.

Runs (/api/2.0/mlflow/runs)​

create, get, update, delete, restore, search, set-tag, delete-tag, log-parameter, log-metric, log-batch, get-metric-history, log-inputs, log-model.

Artifacts (/api/2.0/mlflow/artifacts)​

list, upload, download, delete. Every caller-supplied path is normalised and any .. segment or absolute path is rejected.

Model Registry (/api/2.0/mlflow/registered-models and /api/2.0/mlflow/model-versions)​

registered-models/{create,get,list,search,update,delete,set-tag,delete-tag}, model-versions/{create,get,update,delete,transition-stage,set-tag,delete-tag}, plus the MLflow 2.x extensions:

  • Rename: POST /registered-models/rename
  • Aliases (MLflow 2.6+): POST /registered-models/alias, GET /registered-models/alias, DELETE /registered-models/alias
  • Version search: GET /model-versions/search?filter=...&max_results=...
  • Download URI: GET /model-versions/get-download-uri?name=...&version=...

Prompts, Scorers (MLflow 2.x custom extensions used by the Kirka project)​

Full CRUD exposed under /api/2.0/mlflow/prompts and /api/2.0/mlflow/scorers — see the OpenAPI schema at /v3/api-docs for the exact payloads.

AI Gateway (MLflow 2.8+ style)​

Registration endpoints are implemented: secrets, routes, endpoints, model attachments, tagging. Upstream proxying (actually invoking the target LLM) is not implemented yet — POST /gateway/query/{name} returns 501 ENDPOINT_NOT_IMPLEMENTED with an explicit message. This is tracked as a roadmap item.

Kirka-specific administrative endpoints​

  • GET /api/2.0/kirka/audit/search?max_results=...&page_token=... — admin-only audit trail.
  • POST /api/2.0/kirka/gdpr/hard-delete — admin-only hard delete. Requires the X-Kirka-Confirm-Hard-Delete: <resource_id> header.

Explicitly not implemented​

  • POST /invocations → 501 Not Implemented. Kirka is a tracking server, not a serving runtime. Deploy models on mlflow models serve, Seldon Core, KServe or any other serving stack and point your clients at that endpoint.

Pagination​

Listing and searching endpoints accept max_results (default 1 000, capped at 10 000) and return an opaque next_page_token when more data is available. Re-send the token as the page_token query parameter to fetch the next slice. Tokens are stable under concurrent writes — new rows appear on later pages, deletions simply shrink the scan.
